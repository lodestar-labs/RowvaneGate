Rowvane Gate checks the files organizations actually exchange — multi-record CSV, fixed-width mainframe layouts, nested XML, JSON exports — against one declarative ruleset, and returns structured findings with real line numbers, raw values, and per-rule counts.

One ruleset replaces every hand-written file checker

Declare the shape and the rules once; Gate enforces them for every format.

Every format, one vocabulary

Field rules, cross-field and cross-level comparisons, conditionals, composite uniqueness, parent-vs-children aggregates, row-count bounds — declared once in JSON, enforced identically for CSV, XML, and JSON representations of the same data.

Flat files, done properly

Multi-record-type files with hierarchy reconstructed from record order; header detection and positional mapping; fixed-width slicing; delimiter sniffing; RFC 4180 quoting with escaped quotes and embedded newlines.

Schemas become rulesets

Feed Gate an XSD or a JSON Schema and get an editable native ruleset — types, required fields, enumerations, patterns, and facets become rules that then validate any supported format, not just the schema's own.

Findings, not strings

Every violation carries rule id, severity, entity, field, line number, raw value, and message. Reports add per-entity record counts and per-rule totals that are never truncated — filter, aggregate, diff, render.

An analytical escape hatch

Embedded DuckDB powers column profiling for unfamiliar files and SQL rules — any dataset-level check you can express as a query over the staged file, each returned row becoming a finding.

Streaming end to end

One pass per file. Per-record rules fire as records arrive; dataset rules accumulate and finalize at the end. Memory is bounded by one record subtree, never the file — multi-gigabyte inputs are fine.

You write this…

{
  "name": "trips",
  "shape": {
    "name": "TR",
    "fields": [ { "name": "RecordType" }, { "name": "TripId" },
                { "name": "TotalWeight" } ],
    "children": [ { "name": "HL", "required": true,
      "fields": [ { "name": "RecordType" },
                  { "name": "HaulNo" }, { "name": "Weight" } ] } ]
  },
  "source": { "csv": { "mode": "multiRecord" } },
  "rules": [
    { "entity": "TR", "field": "TripId",
      "check": { "type": "required" } },
    { "entity": "TR",
      "check": { "type": "unique", "fields": [ "TripId" ] } },
    { "entity": "HL", "field": "Weight",
      "check": { "type": "compare", "op": "le",
                 "otherField": "parent.TotalWeight" } },
    { "entity": "TR", "field": "TotalWeight",
      "check": { "type": "aggregate", "childEntity": "HL",
                 "childField": "Weight", "function": "sum",
                 "deviationPercent": 5 } }
  ]
}

…and get this back

{
  "ruleset": "trips",
  "source": "delivery.csv",
  "format": "csv",
  "recordsRead": 8231,
  "recordCounts": { "TR": 412, "HL": 7819 },
  "errorCount": 3,
  "warningCount": 0,
  "findingsByRule": { "R002": 1, "R003": 2 },
  "findings": [
    {
      "ruleId": "R003",
      "severity": "error",
      "entity": "HL",
      "field": "Weight",
      "line": 4711,
      "path": "HL",
      "rawValue": "1200",
      "message": "Value '1200' is not at most
                  parent.TotalWeight (which is '1000')."
    }
  ],
  "valid": false
}

The same ruleset validates the same trips arriving as nested XML or a JSON array — and an XSD or JSON Schema can generate the whole document for you.
