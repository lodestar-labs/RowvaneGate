Rowvane Gate Developer Guide

How the validator is built, why it is built that way, and where to change it.

This guide is for engineers who will maintain and extend Gate. It assumes you can read C# but know nothing about this codebase. Read sections 1–3 in order; the rest can be read on demand.

1. What Gate is

Gate validates data files — multi-record CSV, fixed-width, single CSV, XML, JSON — against a single declarative document per data contract called a ruleset. The ruleset describes the entity hierarchy, the fields, how each format carries them, and the rules the data must satisfy; the engine derives everything else. There is no per-contract code.

The core contract A file streams in → the source reader yields one root-entity subtree at a time → the compiled rule plan evaluates each subtree as it arrives and accumulates dataset state → a structured report comes out. Rule violations are findings (data problems, HTTP 200 with valid: false); a file whose skeleton cannot be read at all (malformed XML, unknown record type, orphaned child record) is a SourceFormatException (HTTP 400). Keeping those two failure classes apart is a design invariant.

Two properties are load-bearing throughout:

  • Streaming. Memory is bounded by one root subtree plus the uniqueness-key sets, never the file.
  • Raw strings. Every value stays exactly what the file said (Dictionary<string, string?>, blank → null). Gate validates what the file actually contains, not a converted interpretation — a dataType check verifies parseability, it never converts.

3. The end-to-end flow

  1. POST /api/validate/{ruleset} receives the upload and stages it to disk — validation streams from the staged file, and sql rules need a real path for DuckDB.
  2. The format is taken from ?format= or inferred from the file extension via each source's Extensions list.
  3. ValidationEngine.ValidateAsync builds a RulePlan from the ruleset (once), then iterates source.ReadAsync(stream, ruleset).
  4. For every yielded root subtree: per-record rules and dataset accumulators run over root.SelfAndDescendants(), then subtree-scoped rules (required children are checked per record; parent-scoped uniqueness and aggregates per completed tree).
  5. After the stream ends: row-count bounds finalize, then sql rules run against the staged file via IAnalyticsRunner.
  6. The report is returned; the staged file is deleted unless Gate:KeepStagedFiles is set.

4. The ruleset model

Core/Rulesets. RulesetDocument holds a Shape (a tree of EntityShape, each with FieldShapes and children), SourceOptions, and a list of Rules. A rule targets an entity (and usually a field), carries a severity, an optional when condition, and a polymorphic RuleCheck:

[JsonPolymorphic(TypeDiscriminatorPropertyName = "type", ...)]
[JsonDerivedType(typeof(RequiredCheck), "required")]
[JsonDerivedType(typeof(TypeCheck), "dataType")]
// ... range, length, regex, inList, compare, unique, aggregate, rowCount, sql
public abstract class RuleCheck
{
    public virtual IEnumerable<string> Validate(RulesetDocument document, EntityShape? entity, Rule rule) { ... }

    [JsonIgnore] public virtual bool RequiresField => true;
}

Three things to know about this model:

  • Checks validate themselves. RulesetDocument.Validate() walks entities, fields, and rules, and delegates check-specific validation to each check's Validate override. It returns every problem, not the first — authors fix a ruleset in one pass. Registration (registry or serializer) refuses invalid documents outright.
  • Field-level checks require an entity. The engine only wires a record rule when rule.Entity is set; the base RuleCheck.Validate therefore rejects field checks without one — a rule that would compile to nothing must not validate.
  • Ids are stable. AssignRuleIds() fills gaps with R001… without touching explicit ids, so findings and suppressions keep meaning across edits.

RulesetSerializer is the single (de)serialization authority: camelCase, case-insensitive, string enums, comments and trailing commas tolerated, and validation + id assignment applied on every deserialize. IRuleset (static abstract Document) lets applications ship compile-time-checked rulesets.

5. The record model

Core/Records/ValidationRecord.cs — the canonical in-flight record every source produces:

public sealed class ValidationRecord
{
    public required EntityShape Entity { get; init; }
    public SourceLocation Location { get; init; }          // real line and/or logical path
    public Dictionary<string, string?> Fields { get; }      // raw values, case-insensitive keys
    public List<ValidationRecord> Children { get; }
    public ValidationRecord? Parent { get; set; }
}

That's the whole contract between readers and the engine. Readers own the mapping from bytes to this shape; the engine never sees a format. SourceLocation carries the parser's real line number when one exists (CSV, XML) and a logical path otherwise (JSONPath for JSON).

6. Findings & the report

Core/Findings/ValidationReport.cs. A Finding is a record: rule id, severity, entity, field, line, path, raw value, message. The report accumulates them with one important wrinkle:

public void Add(Finding finding, int maxPerRule)
{
    // severity counters always increment
    var total = FindingsByRule.TryGetValue(finding.RuleId, out var count) ? count : 0;
    FindingsByRule[finding.RuleId] = total + 1;
    if (total < maxPerRule) { Findings.Add(finding); }   // the LIST is capped; counts never are
}

So a file with 40 000 identical violations reports a readable list and an honest count. Valid is simply "no error-severity findings". Two reserved rule ids: SHAPE (required-child violations) and XSD (native XSD validation findings).

7. The CSV source

Readers/Csv is three cooperating pieces:

CsvParser

A dependency-free RFC 4180 state machine: quoted fields, escaped quotes (""), delimiters and line breaks inside quotes, CR/LF/CRLF endings. It streams rows and reports the line each row starts on, so multi-line quoted values don't shift findings.

Invariant: parser state lives outside the read loop. The parser reads 8 KB buffers; inQuotes, justClosedQuote, previousWasCarriageReturn, and the current field builder must survive buffer refills. Any change that moves state into the loop will pass small-file tests and corrupt large files at buffer boundaries.

CsvSniffer

When the ruleset doesn't pin options, the source reads up to 20 sample lines, scores candidate delimiters (comma/semicolon/tab/pipe) by count-outside-quotes weighted by consistency, decides header presence (first row matching ≥ half the shape's field names), and then replays the sample ahead of the live stream with line numbers offset. Explicit ruleset settings always win — sniffing only fills gaps, so pinned rulesets are deterministic.

Known limitation, documented on purpose: a quoted field spanning the 20-line sample boundary would mis-split during replay. Files that open with multi-line quoted values must pin delimiter in the ruleset. This is a deliberate trade against buffering unbounded input to sniff safely.

CsvValidationSource

Dispatches on source.csv.mode. single maps columns by header or positionally. multiRecord and fixedWidth share the HierarchyAssembler: each record attaches to the most recent record of its parent entity, and the previous root subtree is yielded the moment a new root begins.

Invariant: clear _latest when a new root starts. The assembler tracks the most recent record per entity. When a new root begins, that map must be cleared — otherwise a child record whose parent only exists in the previous tree silently attaches to an already-yielded, already-validated subtree and vanishes from the report (no count, no finding). With the clear, it is what it should be: a structural error naming the line. There is a regression test for this.

8. The XML source

Readers/Xml/XmlValidationSource.cs walks the document with a forward-only async XmlReader and materializes one root-entity subtree at a time. Element names matching child entities recurse; elements or attributes matching shape fields become values; anything else is Skipped — documents may carry extra content without breaking. Real line numbers come from IXmlLineInfo; no custom attributes are needed in source files. DTD processing is prohibited (XXE). An optional source.xml.rootElement wrapper confines matching. XmlExceptions are wrapped into SourceFormatException with the parser's line number.

9. The JSON source

Readers/Json/JsonValidationSource.cs peeks the first significant byte. A top-level array streams element by element via DeserializeAsyncEnumerable — bounded memory for arbitrarily large exports. A top-level object (record array under source.json.rootProperty or a property named like the root entity) is accepted but parses the whole document; the class doc says so and steers users to arrays. Numbers and booleans keep their literal text (GetRawText()), JSON null is a missing value, and locations are JSONPath-style ($[12].hauls[0]).

10. The rule plan

Engine/ValidationEngine.cs, private class RulePlan — the compiled execution plan for one run, resolved before the first record flows. Its constructor sorts every rule into one of five buckets:

BucketMembersEvaluated
Record rulesrequired, dataType, range, length, regex, inList, compare — compiled per entityper record, as it streams
Dataset uniquenessunique with dataset scope — one HashSet per ruleper record (key added as seen)
Subtree rulesunique with parent scope, aggregateper completed root subtree
Row countsrowCount — counters per entityat finalize
SQL rulessqlafter streaming, via IAnalyticsRunner

Required-children checks come from the shape, not from rules: the plan precomputes each entity's required children and verifies them per record (rule id SHAPE). Composite uniqueness keys are built by joining raw field values with an ASCII unit separator (0x1F); records whose key fields are all missing are not checked — presence is required's job.

11. Compiled rules

Engine/CompiledRule.cs. Each per-record rule is prepared once for the hot path: regexes compiled (with a 2-second match timeout), value lists frozen into hash sets with the right comparer, numeric bounds pre-parsed — evaluation is a single delegate call per record:

public Finding? Evaluate(ValidationRecord record)
{
    if (_when is not null && !_when(record)) return null;   // applicability condition
    var message = _evaluate(record);                          // null = pass
    return message is null ? null : new Finding(_rule.Id!, _rule.Severity, ...);
}

Design rules encoded here, worth preserving:

  • Every field check passes on missing values. Only required asserts presence. This keeps all checks composable with optional fields.
  • Invariant culture everywhere. Numbers via decimal.TryParse(NumberStyles.Float, InvariantCulture); dates via DateOnly/DateTime/TimeOnly with optional exact formats.
  • compare resolves field paths. "parent.Field" climbs the record tree (prefix repeatable). Numeric comparison applies only when both sides parse as numbers; otherwise ordinal string comparison.

12. Subtree & dataset checks

EvaluateSubtree runs when a root is complete: parent-scoped uniqueness inspects each sibling group once (triggered by the group's first member), and aggregates compare sum/count/avg/min/max of a child field per parent against the parent's own field — exactly (op) or within deviationPercent. A parent whose own value doesn't parse is skipped (that's a dataType finding's job), and non-count aggregates with zero parseable child values are skipped rather than compared as zero.

Finalize closes the dataset scope: row-count bounds check the per-entity counters (entities that never appeared count as zero, so min works). Then the engine runs sql rules; if no analytics runner is registered or no staged file path was provided, each sql rule degrades to a warning finding instead of failing the run — rulesets stay portable across deployments.

13. XSD import & validation

Schemas/Xsd/XsdImporter.cs compiles the schema set, picks the root global element (explicitly, or the only one), and walks ElementSchemaTypes: complex types with child elements become entities; everything else becomes fields. Constraints map to native rules — minOccurs/use ⇒ required, base type ⇒ dataType, enumeration ⇒ inList (case-sensitive), pattern ⇒ regex (anchored ^(?:…)$, since XSD patterns match whole values), value facets ⇒ range, length facets ⇒ length. Recursive type definitions are cut off via a visiting set. The result is validated like any hand-written ruleset before it leaves the importer.

Schemas/Xsd/XsdValidator.cs is the separate pass-through path: it streams the document through .NET's validating reader and converts every ValidationEventHandler callback into a structured Finding (rule id XSD) with the exception's real line number. Malformed XML becomes one error finding rather than an exception — this endpoint's job is reporting, not refusing.

14. JSON Schema import

Schemas/Json/JsonSchemaImporter.cs handles the widely used subset: type / properties / required / items / enum / pattern / numeric bounds / length bounds / format. Object and array-of-object properties become child entities; the rest become fields. Type unions (["string","null"]) use the first non-null entry. Exclusive bounds accept both spellings: draft 6+ numeric exclusiveMinimum: 5 and draft-4 boolean minimum: 5, exclusiveMinimum: true. Because JSON Schemas rarely name their root, the caller supplies rootEntity (default Record).

15. DuckDB analytics

Analytics/DuckDbAnalytics.cs implements IAnalyticsRunner with an in-memory DuckDB connection per call. For a sql rule it creates CREATE VIEW data AS SELECT * FROM read_csv_auto('…') (or read_json_auto / read_parquet by format), substitutes {file} in the query, and formats each returned row as name=value; name=value — capped at 10 000 rows per rule. Profiling is DuckDB's SUMMARIZE re-shaped into dictionaries. XML staged files are rejected with a clear NotSupportedException; the API surfaces it as 400.

SQL rules are trusted input. Queries come from ruleset authors — the same people who could already declare any rule. The file path is quoted defensively, but the query itself is executed as written, against an in-memory database whose only content is the staged file's view. Do not expose ruleset registration to untrusted callers.

GateInitializer (a hosted service) loads every *.json in the ruleset directory into the registry at startup, skipping — and logging — invalid files rather than refusing to boot. The registry itself is a ConcurrentDictionary keyed case-insensitively; the directory store sanitizes names before using them as file names.

18. Extension points

  • A new source format (Excel, Parquet) is one IValidationSource: a format key, an extensions list for inference, and a ReadAsync that yields root subtrees. Register it as a singleton; the engine and the API pick it up through DI. It must stream.
  • A new check type is a RuleCheck subclass + a [JsonDerivedType] line + either a case in CompiledRule.TryCompile (per-record) or a bucket in RulePlan (stateful). Add its self-validation in Validate.
  • A different analytics engine is one IAnalyticsRunner. The engine takes the first registered runner; absence degrades sql rules to warnings.
  • Compile-time rulesets: implement IRuleset's static abstract Document and register the document at startup — no files, no runtime discovery.

19. Testing

tests/Rowvane.Gate.Tests — NUnit, no mocks, no database. TestData.Trips() builds a compact 3-level hierarchy (Trip → Haul → Sample) shared across the suite. Coverage is organized by seam: ruleset model and serialization round-trips; each CSV mode plus the XML/JSON sources (including line numbers and parent links); engine behavior per check type; schema importers asserting generated rules; DuckDB exercised for real (temp CSV file); regression tests pinning the assembler-reset, entity-requirement, and draft-4 bounds fixes. Run with dotnet test.

For end-to-end verification, drive the API itself — see .claude/skills/verify/SKILL.md in the repo for the exact launch-and-curl recipe.

20. Maintainer gotchas

  • Findings vs exceptions. Data problems are findings; unreadable skeletons are SourceFormatException. Don't blur it — the API's 200/400 split and every consumer's retry logic depend on it.
  • Missing values pass field checks. If a new check fails on null, optional fields break everywhere. Presence is required's job, only.
  • The CSV parser's state lives outside its read loop — see §7. Test any parser change with fields spanning the 8 KB buffer boundary.
  • HierarchyAssembler._latest must be cleared per root — see §7. The failure mode is silent record loss, which no ordinary test notices.
  • Findings caps are list-only. ValidationReport.Add must keep counting past the cap; dashboards rely on exact findingsByRule.
  • Everything matches case-insensitively — entity names, field names, record types, JSON properties, format keys. A new lookup that defaults to ordinal comparison will work in tests and fail on the first real file.
  • dotnet test only rebuilds the test bin. When manually driving the API after library changes, rebuild Rowvane.Gate.Api first or dotnet run --no-build serves stale DLLs.
