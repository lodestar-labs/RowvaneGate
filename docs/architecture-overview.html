Rowvane Gate

Validate any data file — multi-record CSV, fixed-width, XML, JSON — against one declarative ruleset.

Architecture & capabilities overview · github.com/lodestar-labs/RowvaneGate · .NET 10 · BSL 1.1

1. The core idea

Every organization that receives files eventually hand-writes the same program: parse the file, walk its structure, check the fields, check the totals, and produce some report of what was wrong — then writes it again, slightly differently, for the next feed and the next format. Gate replaces that entire class of one-off checkers with a single declarative ruleset per data contract. The ruleset is the only thing you author (or generate from an XSD / JSON Schema). Everything else is derived from it:

You declare (once)Gate derives
Entity hierarchy, fields, per-format carriage (record types, positions, wrappers), and the rules the data must satisfy Format parsing expectations for CSV / fixed-width / XML / JSON · compiled rule evaluation · hierarchy reconstruction and shape checks · uniqueness and row-count accumulators · aggregate reconciliation · the structured validation report
The core contract: a file streams in → records flow through the compiled rule plan one root subtree at a time → a structured report streams out: every violation with rule id, severity, entity, field, real source line, and the raw value. A file whose skeleton is broken (malformed XML, unknown record type, orphaned child record) fails fast with the offending line; everything else is findings, never exceptions.

2. System architecture

Six projects in a strict layering. The golden rule: Core has no dependencies and defines the models and contracts; everything else implements them. New formats are new IValidationSource implementations; the analytics tier is optional and plugs in behind IAnalyticsRunner.

3. The validation pass

Each validation is one streaming pass. The source reader yields completed root subtrees; the engine evaluates three scopes of rules against them without ever holding the file:

Rule compilation

Rulesets are interpreted once, before the first record flows. Each rule's configuration is parsed into a compiled form — regexes compiled with a timeout, value lists frozen into hash sets, numeric bounds pre-parsed — and evaluation reduces to one delegate call per record. No reflection and no re-interpretation of JSON in the hot path. The compiled plan also resolves the shape's required-child structure and sets up the uniqueness and row-count accumulators up front.

4. Flat files are the hard part — and the point

XML and JSON carry their own structure; flat files don't, and they are what suppliers actually send. Gate's CSV source covers the three shapes that occur commercially:

ModeWhat it handles
single One record type per file. Columns map by header row — detected automatically when the first row matches the shape's field names — or positionally by field order.
multiRecord Every line announces its record type in a discriminator column (bank files, mainframe extracts, scientific exchange formats). The hierarchy is reconstructed from record order: each record attaches to the most recent record of its parent entity; a completed root subtree is yielded as soon as the next root begins. A child record with no parent in the current tree is a structural error — records are never silently attached to an already-validated tree.
fixedWidth Like multiRecord, but fields are sliced by each field's declared start/length. The layout document is transcribed into the shape once, instead of living in every consumer's substring arithmetic.

Underneath sits a dependency-free RFC 4180 parser — quoted fields, escaped quotes, delimiters and line breaks inside quotes, CR/LF/CRLF endings — that reports the physical line each row starts on, so findings point at the right place even for multi-line values. Delimiters are sniffed across comma/semicolon/tab/pipe when the ruleset doesn't pin one; explicit settings always win, so behavior is deterministic once an author pins the options.

5. Schema import: meet contracts where they are

Many data contracts already exist as an XSD or a JSON Schema. Gate imports either into a native, editable ruleset: the element hierarchy becomes the shape; minOccurs and required become required rules; enumerations become inList; patterns become regex; value and length facets become range and length; base types become dataType. The payoff is the part XSD alone can never do: import the schema once, then validate CSV or JSON representations of the same data against the identical rules. For consumers whose contract is strictly the XSD, native pass-through validation (with the parser's real line numbers as structured findings) is also built in.

6. The analytical tier

Some checks span the whole dataset in ways per-record rules can't express. Gate embeds DuckDB for two capabilities, both optional:

  • SQL rules — the ruleset's sql check runs a query against the staged file (exposed as the view data); every returned row becomes a finding with the row's columns formatted into the message. Deduplication windows, ratio checks, statistical outliers — if you can query it, you can gate on it.
  • Profiling — POST /api/profile returns per-column statistics (types, ranges, null percentages) for a quick picture of an unfamiliar file before writing rules for it.

Deployments without the analytics package still run every declarative rule; sql rules degrade to warning findings, so rulesets stay portable.
